Delivery Flow

変更反映の流れ

PR to production

  1. 01
    PR
    Pull Request

    変更をレビュー可能な単位に分け、差分と目的を明確にします。

  2. 02
    CI
    GitHub Actions CI

    必須ファイル、依存関係、API構文、PRガードレールを検証します。

  3. 03
    DEP
    Worker deploy

    mainへの反映後、対象変更に応じてCloudflare Workerへデプロイします。

  4. 04
    TEST
    Production checks

    本番URLとAPIエンドポイントの期待ステータスを確認します。

  5. 05
    VERIFY
    Production verification

    必要に応じて手動確認し、Issueや日次ログへ結果を残します。

Production Check

本番確認

Expected checks

FE https://sre-lab.pages.dev/ 200

公開フロントエンドがHTTP 200を返すことを確認します。

API /api/health 200

Primary uptime checkです。Availability SLIの中心として扱います。

API /api/status 200

サービス状態とアクティブなエンドポイントを確認します。

API /api/fallback 200

fallback demoが期待通り利用可能であることを確認します。

EXP /api/error 500 expected

意図的なエラーデモです。本番確認では期待値として検証します。

Failure Handling

失敗時の対応

Rollback policy

CI failure

PR上で検証が失敗した場合は、mainへ入れずに差分、依存関係、API構文、ガードレールを修正します。

Deploy failure

Worker deployが失敗した場合は、Secrets、wrangler設定、直近のAPI変更、Actionsログを確認します。

Production check failure

本番反映後の確認に失敗した場合は、影響範囲を確認し、必要に応じてrevertまたは前回安定版へ戻します。
